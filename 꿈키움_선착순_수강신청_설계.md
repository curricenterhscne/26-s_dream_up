## 1. 가장 중요한 기술 과제 2가지

이 시스템의 난이도는 사실상 아래 두 가지에 모두 들어있습니다. 나머지는 평범한 CRUD입니다.

### (A) 동시성 — 정원 초과 방지 (Race Condition)
선착순이라 마감 직전에 수십 명이 같은 강좌를 동시에 누릅니다.
"현재 인원을 SELECT로 세고 → 15명 미만이면 INSERT" 방식은 **반드시 정원을 초과**합니다(두 요청이 동시에 14명을 읽고 둘 다 INSERT).

**해결: 클라이언트가 직접 INSERT하지 못하게 하고, 모든 신청을 단일 원자적(atomic) RPC로 처리.**
강좌 행을 잠그고 `enrolled_count`를 조건부 증가시키는 한 문장으로 정원을 보장합니다. (§4 SQL 참조)

### (B) 보안 — 로그인 없는 시스템의 개인정보 보호
로그인이 없으면 클라이언트에 박히는 **anon key는 사실상 공개**됩니다.
따라서 테이블을 클라이언트에서 직접 SELECT/INSERT하게 두면 **누구나 전체 신청자 명단(이름·학번·휴대폰)을 긁어갈 수 있습니다.**

**해결:**
- 모든 테이블 **RLS(Row Level Security) 활성화 + 직접 접근 전면 차단**.
- 신청/취소/조회는 전부 **`SECURITY DEFINER` RPC** (또는 Edge Function) 경유.
- 개인정보(휴대폰·이름)는 **클라이언트로 절대 반환하지 않음**. 본인 확인은 "학번+휴대폰 일치" 같은 입력값 대조로만.

### (C) 시간 게이팅 — 6.25 18:00 자동 오픈
**클라이언트 시계를 믿으면 안 됨**(시계 조작·미리 신청 가능). 오픈 판정은 **서버 시각 기준**으로만.
- `settings` 테이블(또는 상수)에 `open_at = 2026-06-25 18:00 KST`, 신청기간(1차/2차) 저장.
- `apply_course` RPC가 `now()`와 비교해 기간 밖이면 거부.
- 프론트는 화면 전환(카운트다운 → 신청 모드)을 위해 **서버 시각/`open_at`을 RPC로 받아** UI를 바꾸되, **실제 차단은 RPC가** 한다.
- ⚠️ 타임존: Supabase Postgres는 UTC. `'2026-06-25 18:00+09'`로 저장(= 09:00 UTC). 클라이언트 표시는 KST로 포맷.

## 4. 동시성 안전 신청 RPC (이 시스템의 심장)

```sql
CREATE OR REPLACE FUNCTION apply_course(
  p_code text, p_school text, p_student_no text,
  p_name text, p_phone text
) RETURNS jsonb
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE
  v_phone text := regexp_replace(p_phone, '[^0-9]', '', 'g');
  v_updated int;
BEGIN
  -- 0) 신청기간 체크 (서버 시각 기준). settings.open_at = 2026-06-25 18:00+09
  IF now() < (SELECT open_at FROM settings LIMIT 1) THEN
    RETURN jsonb_build_object('ok',false,'error','not_open');   -- 아직 오픈 전
  END IF;
  -- (선택) 1차/2차 기간 밖이면 'period' 반환

  -- 1) 이미 신청한 학생인가? (1인 1강좌)
  IF EXISTS (SELECT 1 FROM enrollments
             WHERE status='active'
               AND (school=p_school AND student_no=p_student_no
                    OR phone=v_phone)) THEN
    RETURN jsonb_build_object('ok',false,'error','duplicate');
  END IF;

  -- 2) 신청 가능 강좌인지(DC26B054 등 제외)
  IF NOT EXISTS (SELECT 1 FROM courses WHERE code=p_code AND is_registerable) THEN
    RETURN jsonb_build_object('ok',false,'error','not_registerable');
  END IF;

  -- 3) 정원 원자적 확보: 행 잠금 + 조건부 증가 (핵심)
  UPDATE courses
     SET enrolled_count = enrolled_count + 1
   WHERE code = p_code
     AND is_registerable = true
     AND is_closed_manual = false
     AND enrolled_count < capacity
  RETURNING 1 INTO v_updated;

  IF v_updated IS NULL THEN
    RETURN jsonb_build_object('ok',false,'error','full');  -- 마감
  END IF;

  -- 4) 신청 기록 INSERT (유니크 위반 시 정원 롤백)
  BEGIN
    INSERT INTO enrollments(course_code,school,student_no,name,phone)
    VALUES (p_code,p_school,p_student_no,p_name,v_phone);
  EXCEPTION WHEN unique_violation THEN
    UPDATE courses SET enrolled_count = enrolled_count - 1 WHERE code=p_code;
    RETURN jsonb_build_object('ok',false,'error','duplicate');
  END;

  RETURN jsonb_build_object('ok',true);
END $$;
```

핵심은 **3번 `UPDATE ... WHERE enrolled_count < capacity RETURNING`**:
Postgres가 그 행을 잠그고 조건을 보장하므로, 동시에 100명이 들어와도 `capacity`를 넘는 순간부터는 `v_updated`가 NULL → 'full' 반환. 정원 초과가 **구조적으로 불가능**해집니다.

### 취소 RPC (마감 전까지 본인)
```sql
CREATE OR REPLACE FUNCTION cancel_enrollment(
  p_school text, p_student_no text, p_phone text
) RETURNS jsonb
LANGUAGE plpgsql SECURITY DEFINER AS $$
DECLARE v_code text;
BEGIN
  -- 본인확인: 학교+학번+휴대폰 일치하는 active 신청 찾기
  SELECT course_code INTO v_code FROM enrollments
   WHERE status='active' AND school=p_school
     AND student_no=p_student_no
     AND phone=regexp_replace(p_phone,'[^0-9]','','g')
   FOR UPDATE;
  IF v_code IS NULL THEN RETURN jsonb_build_object('ok',false,'error','notfound'); END IF;

  UPDATE enrollments SET status='cancelled'
   WHERE status='active' AND school=p_school AND student_no=p_student_no;
  UPDATE courses SET enrolled_count = greatest(enrolled_count-1,0) WHERE code=v_code;
  RETURN jsonb_build_object('ok',true);
END $$;
```
> "마감 전까지"의 의미: 정원이 찼어도 본인 취소는 가능해야 하므로, 취소는 신청기간 종료 전이면 허용(정원과 무관). 취소하면 그 강좌에 자리가 생겨 **자동 재오픈**됩니다.

### 조회 RPC (개인정보 최소반환)
- `get_course_status()` → 각 강좌의 `code, enrolled_count, capacity, 남은자리, 마감여부`만 반환. **명단·개인정보 반환 금지.**
- `get_my_enrollment(school, student_no, phone)` → 본인 1건의 강좌명·상태만 반환.

### RLS
```sql
ALTER TABLE courses ENABLE ROW LEVEL SECURITY;
ALTER TABLE enrollments ENABLE ROW LEVEL SECURITY;
-- 정책을 만들지 않으면 anon은 직접 SELECT/INSERT 불가.
-- 잔여석 표시에 필요한 최소 컬럼은 RPC(get_course_status)로만 노출.
```

## 5. 화면 구성 (학생용)

**오픈 전(6.25 18:00 이전):** 안내 페이지 + 강좌 카탈로그를 그대로 보여주되, 상단에 **"신청 오픈까지 D-카운트다운"** 배너, 신청 버튼은 비활성. (현재 안내 HTML과 거의 동일한 모습)
**오픈 후:** 같은 페이지가 자동으로 **신청 모드**로 전환 — 카드마다 잔여석·신청 버튼 활성화.

1. **강좌 목록 / 실시간 잔여석** — 안내 HTML의 카드 UI 재사용 + 강좌별 `남은자리 N / 마감` 배지. 5~10초 폴링 또는 Supabase Realtime 구독. **`DC26B054`는 '보령시 별도 모집' 안내 배지 + 신청 버튼 없음**(기존 🔒 처리 패턴 재사용).
2. **신청 폼** — 학교·학번·이름·휴대폰 입력 → `apply_course` 호출 → 결과 토스트(성공/마감/중복신청/오픈전). 폼에 안내문구 필수 노출:
   > "휴대폰 번호는 수강 안내 및 본인 확인 용도로 사용되니 **정확하게 입력**해 주세요."
3. **내 신청 확인/취소** — 학교+학번+휴대폰 입력 → 본인 신청 1건 표시 → 마감 전 취소 버튼.

UX 주의:
- 마감된 강좌는 버튼 비활성 + 표시. 휴대폰은 `010-0000-0000` 입력받아 숫자만 정규화 저장.
- 중복 클릭 방지(버튼 더블서밋 락). 오픈 직후 트래픽 급증 대비 버튼 로딩 상태 처리.

## 7. 배포 파이프라인
1. **Supabase 프로젝트 생성** → `supabase/migrations/`에 §3·§4 SQL을 마이그레이션으로 저장. `settings.open_at = '2026-06-25 18:00+09'` 시드.
2. **시드**: HTML `DATA` 배열 파싱 → `courses` upsert(전 강좌 capacity=15, **`DC26B054`는 `is_registerable=false`**).
3. **학생 화면 = 기존 GitHub Pages 레포(`26-s_dream_up`)에 배포**: 현재 `index.html` 백업 커밋 → 신청 기능 포함 새 `index.html`로 교체. anon key는 클라이언트에 들어가도 되지만, **RLS·RPC로 보호되어 있어야** 안전(키 자체는 공개 전제).

## 10. 사전 테스트 / 드레스 리허설 계획 (6.25 18:00 전)

핵심 아이디어: **오픈 시각이 `settings.open_at` 한 값으로 제어**되므로, 이 값만 바꾸면 언제든 "18:00 오픈"을 흉내 낼 수 있습니다. 단, **실제 신청 데이터와 섞이면 안 되므로 분리**가 필수입니다.

### 10-1. 테스트 환경 = 운영과 동일한 단일 프로젝트
무료 프로젝트 1개에서 **리허설 → 초기화 → 실제 접수**를 모두 진행합니다(잔여 슬롯 사용 안 함).
이 방식이 안전한 이유: **본번 오픈(6.25 18:00) 전에는 테이블에 '진짜 신청'이 존재하지 않으므로, 그 안의 모든 데이터는 테스트 데이터** → 리허설 후 전량 삭제해도 잃을 운영 데이터가 없습니다.

안전을 위해 두 가지 잠금장치를 둡니다(§10-5):
- **시간 잠금**: 초기화는 `now() < open_at`일 때만 허용 → 본번 시작 후에는 어떤 경로로도 실수로 데이터가 지워지지 않음.
- **검증 쿼리**: 초기화 직후 신청 0건·카운트 0·`open_at` 정상인지 눈으로 확인.

### 10-2. 기능 테스트 체크리스트
| # | 시나리오 | 기대 결과 |
|---|---|---|
| 1 | 정상 신청 1건 | 성공, `enrolled_count +1` |
| 2 | 같은 학번으로 2번째 강좌 신청 | `duplicate` 거부 (1인 1강좌) |
| 3 | 같은 휴대폰으로 다른 학번 신청 | `duplicate` 거부(보조 차단) |
| 4 | 정원 꽉 찬 강좌 신청 | `full` 거부 |
| 5 | 신청 취소 → 같은 학생 재신청 | 취소 성공 → 재신청 성공(부분 유니크 인덱스 동작) |
| 6 | 취소로 자리 난 마감 강좌에 다른 학생 신청 | 성공(자동 재오픈) |
| 7 | `open_at` 미래로 둔 상태에서 신청 | `not_open` 거부(시간 게이팅) |
| 8 | `DC26B054` 신청 | `not_registerable` 거부 |
| 9 | 잘못된 본인정보로 취소 시도 | `notfound` 거부 |

> 테스트용 강좌 하나는 정원을 일부러 `2~3`으로 낮춰 마감(4·6번)을 빠르게 검증.

### 10-3. 동시성 / 부하 테스트 (가장 중요)
- **목표:** 정원 15 강좌에 **동시 N요청**을 던졌을 때 ① 정확히 15건만 성공(초과 0건), ② 응답시간·에러율이 허용 범위인지.
- **도구:** `k6`(권장) / `Artillery` / 또는 Node 스크립트(`Promise.all`로 동시 RPC 수백~수천 발사).
- **단계적 부하:** 동시 100 → 300 → 700 → 1,500으로 올리며 각 단계의 **성공/거부/타임아웃 비율, p95 응답시간**을 기록 → 우리 환경의 실제 한계선을 찾음.
- **검증 핵심:** 어떤 부하에서도 **오버셀(정원 초과)은 0**이어야 함(원자적 UPDATE 덕). 부하가 높으면 "타임아웃→재시도"는 생겨도 데이터는 항상 정확.

### 10-4. 드레스 리허설 (실전 시뮬레이션)
1. 테스트 환경에서 `open_at`을 "지금부터 5분 뒤"로 설정.
2. 동료·테스터 여러 명이 각자 기기로 접속 → **카운트다운 화면** 확인.
3. 시각이 되면 **자동으로 신청 모드 전환**되는지, 동시에 같은 인기 강좌를 눌렀을 때 선착순·마감이 정상인지 육안 확인.
4. 취소→재신청, 마감 강좌 표시, 휴대폰 안내문구, `DC26B054` 별도모집 표시까지 점검.

### 10-5. 테스트 데이터 삭제 & 본번 초기화

리허설이 끝나면 **테스트 데이터를 깨끗이 지우고** 본번 상태로 되돌립니다. 같은 프로젝트라 **안전장치(시간 잠금 + 검증)** 가 핵심입니다.

#### (a) 초기화 SQL — Supabase SQL Editor에서 service_role로 실행
```sql
-- ⚠️ 본번 시작 후 실수 방지: open_at 이후엔 아예 실행 거부
DO $$
BEGIN
  IF now() >= (SELECT open_at FROM settings LIMIT 1) THEN
    RAISE EXCEPTION '초기화 거부: 이미 신청 오픈 시각(open_at) 이후입니다.';
  END IF;

  -- 1) 신청 데이터 전량 삭제
  TRUNCATE TABLE enrollments;
  -- 2) 강좌 카운터/수동마감 리셋
  UPDATE courses SET enrolled_count = 0, is_closed_manual = false;
  -- 3) 오픈 시각을 본번으로 복원
  UPDATE settings SET open_at = '2026-06-25 18:00+09';
  -- 4) (정원을 테스트로 낮췄었다면) 전 강좌 정원 15 복원
  UPDATE courses SET capacity = 15;  -- DC26B054는 어차피 is_registerable=false
END $$;
```
> `TRUNCATE`는 전체 비우기라 빠르고 시퀀스도 정리됩니다. 신청 0건 전제(본번 전)라서 전량 삭제가 정답.

#### (b) (선택) 관리자 페이지 버튼용 RPC — 같은 시간 잠금 적용
SQL Editor 대신 관리자 화면에서 버튼으로 누르고 싶다면, **service_role로만** 호출되는 RPC로 감쌉니다. 실수 방지를 위해 **확인 문구**까지 받습니다.
```sql
CREATE OR REPLACE FUNCTION admin_reset_test_data(p_confirm text)
RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  IF p_confirm <> 'RESET-실행확인' THEN
    RETURN jsonb_build_object('ok',false,'error','confirm_mismatch');
  END IF;
  IF now() >= (SELECT open_at FROM settings LIMIT 1) THEN
    RETURN jsonb_build_object('ok',false,'error','already_open');  -- 본번 후 차단
  END IF;
  TRUNCATE TABLE enrollments;
  UPDATE courses SET enrolled_count = 0, is_closed_manual = false;
  UPDATE settings SET open_at = '2026-06-25 18:00+09';
  RETURN jsonb_build_object('ok',true);
END $$;
-- 일반(anon) 사용자 호출 차단, service_role만 실행 가능하게:
REVOKE EXECUTE ON FUNCTION admin_reset_test_data(text) FROM anon, authenticated;
```
> 이 RPC는 관리자 페이지(Vercel, service_role 키)에서만 호출. 학생 화면의 anon 키로는 실행 불가.

#### (c) 초기화 검증 쿼리 (반드시 눈으로 확인)
```sql
SELECT
  (SELECT count(*) FROM enrollments)                       AS 신청건수,      -- 0 이어야 함
  (SELECT count(*) FROM courses WHERE enrolled_count <> 0) AS 카운트비정상,  -- 0 이어야 함
  (SELECT open_at FROM settings LIMIT 1)                   AS 오픈시각;      -- 2026-06-25 18:00+09
```

## 11. 무료 티어 동시 접속 한계 분석 (Supabase Free + GitHub Pages)

> 수치는 2026년 기준이며 **수시로 바뀝니다. supabase.com/pricing에서 본번 전 재확인** 권장.

### 11-1. 무료 티어 핵심 한도
- **컴퓨트:** 공유 인스턴스(RAM 약 500MB, 공유 CPU) — 동접 스파이크 시 가장 먼저 병목.
- **DB 커넥션:** 직접 60 / **풀러(pooler) 200**. (RPC는 풀러를 거치므로 200이 동시 처리 한계의 기준)
- **API 요청:** 횟수 자체는 사실상 무제한(이그레스 5GB/월 내) — RPC 응답 JSON은 매우 작아 신청 수만 건이라도 5GB와 무관.
- **Realtime:** **동시 200 연결**, 메시지 200만/월. → **여기가 함정.**
- **휴면:** 7일 무활동 시 일시정지(본번엔 활성이라 무관, 단 사전 웜업 필요).

### 11-2. 우리 앱의 실제 부하 구조
- **페이지 로딩**: GitHub Pages(CDN). Supabase와 무관하며 수천 명도 거뜬.
- **잔여석 표시 / 신청 / 취소**: 전부 작은 RPC HTTP 요청. 신청 RPC는 **해당 강좌 행만 잠그는 수 ms짜리 짧은 트랜잭션**.
- **총 작업량 자체는 사소**: 좌석 총합 88×15 = **1,320건**. Postgres엔 우스운 양.

### 11-3. 진짜 위험은 "18:00:00 동시 쇄도(thundering herd)"
총량이 아니라 **오픈 정각에 모두가 동시에 누르는 순간 스파이크**가 문제입니다.

- **데이터 정합성은 부하와 무관하게 항상 안전** — 원자적 UPDATE라 아무리 몰려도 **정원 초과(오버셀)는 구조적으로 0**. 최악의 경우 일부 사용자가 "지연/타임아웃→재시도"를 겪을 뿐, 잘못된 신청은 발생하지 않음.
- **공유 컴퓨트 한계상**, 정각에 **수백 명 수준**이면 무료로 대체로 견딤. **1,000~수천 명이 같은 초에 몰리면** 지연·일부 503/타임아웃이 생겨 재시도가 필요할 수 있음(충남 전역 대상이면 이 구간 가능성 있음).

### 11-4. ⚠️ Realtime은 쓰지 말 것 (무료 200 동접)
실시간 잔여석을 Realtime 구독으로 구현하면, 학생 1,000명 접속 시 1,000 동시연결이 되어 **200 한도를 즉시 초과 → 채널 조인 거부**.
→ **무료 티어에서는 Realtime 대신 "폴링"으로 잔여석 갱신.** (10~15초 간격 + 클라이언트마다 무작위 지터를 줘서 동시 폴링이 한 틱에 몰리지 않게.)

### 11-5. 무료 티어로 스파이크를 버티게 하는 장치
1. **Realtime 금지 → 폴링 + 지터**(위 11-4).
2. **`get_course_status`를 가볍게**: 강좌별 잔여석만 반환하는 단일 집계. 잠깐의 staleness 허용.
3. **클라이언트 보호**: 버튼 더블서밋 락 + 타임아웃 시 지수 백오프 자동재시도 + "신청이 몰리고 있어요, 잠시 후 자동 재시도" UI.
4. **오픈 직후 0~2초 무작위 지연 후 버튼 활성화**: 정확히 같은 순간의 쇄도를 평탄화(선착순 공정성은 DB 도착순으로 유지되므로 영향 미미). — 운영 정책상 허용되면 강력 추천.
5. **본번 웜업**: 18:00 직전 미리 요청을 보내 인스턴스를 깨워둠(콜드스타트 방지).

### 11-6. 결론 & 권고
- **정합성(오버셀 방지)** 은 무료 티어로도 100% 보장됨. 걱정할 건 **속도/체감 안정성**뿐.
- **예상 동접이 수백 명 이하** → 무료로 충분(위 장치 적용 시).

  - **(저비용 보험) 출시 달만 Supabase Pro($25/월)로 업그레이드** 후 종료되면 다시 무료로. Pro는 컴퓨트·커넥션이 커지고, Realtime 500, 일일 백업, 휴면 없음 → 일회성 고부하 이벤트에 가장 안전한 한 수.
